## 🏛️ 组织架构设计

### 核心理念

```
组织 = 角色 + 流程 + 记忆
人员 = 角色的临时承载者
```

**人员流动**：
```
新人加入 → 阅读文档 → 承担角色 → 工作 → 离开
    ↓
组织记忆保留，新人继续承载
```

### 标准角色定义

| 角色代码 | 角色名称 | 职责 | 交接文档 |
|---------|---------|------|---------|
| **LEADER** | 掌印大人（最高管理者） | 战略决策、最终验收 | `docs/LEADER_HANDOVER.md` |
| **COORDINATOR** | 综合管理者（天策府·左护法） | 全流程执行 + 战略/决策/产品/运营/设计/审查/验收/团队管理 | `docs/COORDINATOR_HANDOVER.md` |
| **BUILDER** | 建设者（神机营·都统 + 镇抚司·右护法） | 全流程工作（开发-审查-测试-优化） | `docs/BUILDER_HANDOVER.md` |
| **NEWCOMER** | 新人 | 学习、试跑、成长 | `docs/NEWCOMER_GUIDE.md` |

**角色映射**（猎脉项目）：
- LEADER = 掌印大人（Liman）/ 司命大人
- COORDINATOR = 天策府（左护法）
- BUILDER = 神机营（都统）+ 镇抚司（右护法）
- NEWCOMER = 新加入 Agent

**核心原则**：
- 掌印大人 = 司命大人 = Liman，最高管理者，战略决策和最终验收
- 天策府（左护法）= **最高级别综合管理部门**，既具备全流程执行能力（开发-审查-测试-优化），又承担战略、决策、产品、运营、设计、审查、最终验收、团队管理等额外职责
- 镇抚司（右护法）= 执行团队，负责**全流程工作**（开发-审查-测试-优化）
- 神机营（都统）= 执行团队，负责**全流程工作**（开发-审查-测试-优化）
- **没有专门的"审查部门"**，审查是全流程的一部分

## 🔄 人员流动机制

### 场景 1: 新人加入

**流程**：
```
1. 阅读新人引导文档（30 分钟）
2. 选择角色（COORDINATOR/BUILDER）
3. 阅读角色交接手册（30 分钟）
4. 阅读项目记忆库（30 分钟）
5. 领取任务，开始工作
```

**文档**：
- `docs/NEWCOMER_GUIDE.md` - 新人引导
- `docs/[ROLE]_HANDOVER.md` - 角色交接手册
- `docs/PROJECT_MEMORY.md` - 项目记忆库

### 场景 2: 角色交接

**流程**：
```
1. 提前通知（计划交接）或 立即触发（紧急交接）
2. 更新交接文档（补充最新经验）
3. 通知组织成员
4. 继任者阅读文档
5. 并行工作（可选）
6. 正式交接
7. 发布公告
```

**文档**：
- `docs/COORDINATOR_HANDOVER.md` - 协调者交接手册
- `docs/PROJECT_MEMORY.md` - 项目记忆库（更新最新经验）

### 场景 3: 人员离开

**流程**：
```
1. 更新交接文档（补充遗漏知识）
2. 整理工作文件
3. 通知组织成员
4. 指定继任者（或公开招募）
5. 完成交接
6. 离开
```

**检查清单**：
- [ ] 交接文档已更新
- [ ] 工作文件已整理
- [ ] 继任者已指定
- [ ] 组织成员已通知
- [ ] 权限已移交

## 🎯 核心流程

### 流程 1: 任务流转

```
需求 → 分解 → 分配 → 执行 → 验收 → 闭环
  ↓      ↓      ↓      ↓      ↓      ↓
LEADER  LEADER COORDINATOR BUILDER BUILDER  COORDINATOR
```

**详细说明**：
1. **需求** - LEADER 提出战略目标
2. **分解** - LEADER 分解为具体任务
3. **分配** - COORDINATOR 分配给 BUILDER
4. **执行** - BUILDER 执行全流程工作（开发-审查-测试-优化）（5 分钟闭环）
5. **验收** - BUILDER 自我审查质量，COORDINATOR 监控进度，LEADER 最终验收
6. **闭环** - COORDINATOR 更新状态，通知 LEADER

### 流程 2: 知识沉淀

```
工作 → 记录 → 整理 → 归档 → 传承
  ↓      ↓      ↓      ↓      ↓
BUILDER BUILDER COORDINATOR 记忆库 新人
```

**详细说明**：
1. **工作** - BUILDER 完成全流程工作
2. **记录** - BUILDER 记录经验教训
3. **整理** - COORDINATOR 整理为文档
4. **归档** - 存入项目记忆库
5. **传承** - 新人阅读学习

### 流程 3: 决策流转

```
问题 → 讨论 → 决策 → 记录 → 执行 → 回顾
  ↓      ↓      ↓      ↓      ↓      ↓
任何成员 全体 LEADER 记忆库 全体 LEADER
```

**详细说明**：
1. **问题** - 任何成员发现问题
2. **讨论** - 全体成员讨论方案
3. **决策** - LEADER 做出最终决策
4. **记录** - 存入决策记录
5. **执行** - 全体成员执行
6. **回顾** - LEADER 回顾效果

## 🚨 紧急情况处理

### 场景 1: 关键角色突然掉线

**症状**：
- LEADER/COORDINATOR 不回复
- 任务无人分配/验收
- 组织停滞

**立即行动**：
```
1. 剩余成员中指定临时继任者
   - LEADER 掉线 → 指定临时 LEADER
   - COORDINATOR 掉线 → 指定临时 COORDINATOR

2. 临时继任者阅读交接手册（30 分钟）

3. 发布公告：
   "@全体 [角色] 掉线，我是临时 [角色] [名字]，已接管职责"

4. 按清单执行职责

5. 等待正式继任者
```

### 场景 2: 多人同时掉线

**症状**：
- 多个关键角色不回复
- 组织几乎停滞

**立即行动**：
```
1. 剩余成员中能力最强者担任临时 LEADER

2. 临时 LEADER 分配角色：
   - 自己：LEADER + COORDINATOR
   - 其他人：BUILDER + REVIEWER

3. 优先保证核心业务运转

4. 招募新人补充

5. 逐步恢复正常
```

### 场景 3: 组织分裂

**症状**：
- 成员分成多派
- 决策无法执行
- 组织内耗

**立即行动**：
```
1. LEADER 召集全体成员会议

2. 明确组织目标（回归初心）

3. 重新定义角色和职责

4. 必要时重组组织

5. 记录教训到记忆库
```

## 💬 常见问题

### Q1: 小组织需要这套体系吗？

**A**: 需要！越早建立越好。
- 3 人组织：简单文档即可
- 10 人组织：需要完整体系
- 100 人组织：需要自动化工具

**建议**：从第一天就开始记录决策和经验。

### Q2: 文档太多，维护不过来怎么办？

**A**: 文档在精不在多。
- 核心文档（5 个）：必须维护
- 参考文档：按需更新
- 过时文档：归档或删除

**原则**：文档有人用才维护，没人用就删除。

### Q3: 新人不愿意读文档怎么办？

**A**: 
1. 文档要简洁（30 分钟读完）
2. 文档要有用（解决实际问题）
3. 设置考核（必须通过）
4. 老人带新人（言传身教）

### Q4: 如何保证文档及时更新？

**A**:
1. 责任到人（指定维护者）
2. 定期检查（每周检查）
3. 纳入考核（更新情况计入绩效）
4. 工具辅助（自动提醒）

## 📞 实施指南

### 第 1 步：建立核心文档（1 小时）

```
1. 创建 ORGANIZATION.md（组织总览）
2. 创建 NEWCOMER_GUIDE.md（新人引导）
3. 创建 TASK_BOARD.md（任务看板）
4. 创建 PROJECT_MEMORY.md（项目记忆库）
```

### 第 2 步：定义角色（1 小时）

```
1. 定义组织需要的角色
2. 为每个角色创建交接手册
3. 分配成员到角色
4. 通知全体成员
```

### 第 3 步：建立流程（1 小时）

```
1. 定义任务流转流程
2. 定义知识沉淀流程
3. 定义决策流转流程
4. 通知全体成员
```

### 第 4 步：试运行（1 周）

```
1. 按新流程工作 1 周
2. 每日收集反馈
3. 每日优化流程
4. 周末回顾总结
```

### 第 5 步：正式运行

```
1. 发布正式运行公告
2. 持续监控健康度
3. 每月回顾优化
4. 每季度大版本更新
```
