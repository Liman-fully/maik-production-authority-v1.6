## 📌 核心原则（必须遵守）

### 1. 三源验证
汇报前必须验证三个信息源：
```
1. 任务看板 (.task-board.md)
2. git 提交记录 (git log)
3. 代码文件 (ls/find)

三者一致 → 可汇报
三者不一致 → 先核查再汇报
```

### 2. Git 为准
- 一切以 git 提交记录为准
- 文档必须与代码同步
- 提交信息必须包含任务 ID

### 3. 责任到人
- 每个任务必须有明确的实际执行者
- 子代理任务：创建者=标注者=责任承担者

## 📋 工作流程

### 开始工作
```bash
# 1. 获取最新代码
git pull origin master

# 2. 查看任务看板
cat .task-board.md

# 3. 领取任务（更新状态为"进行中"）
```

### 执行任务
```bash
# 1. 创建功能分支
git checkout -b feature/任务 ID

# 2. 执行开发

# 3. 提交代码（小步提交）
git add .
git commit -m "feat(任务 ID): 简短描述"
git push origin feature/任务 ID
```

### 完成任务
```bash
# 1. 更新任务看板（状态改为"已完成"）
# 2. 更新今日进展（docs/TODAY_PROGRESS.md）
# 3. 合并到主分支（PR 或直接 merge）
# 4. 汇报完成
```

## 📝 提交规范

### 提交信息格式
```
<类型>(<任务 ID>): <简短描述>

<详细说明（可选）>

交付物:
- [文件 1]
- [文件 2]

关联任务：[任务 ID]
```

### 提交类型
| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | feat(DEV-005): 实现人才推荐算法 |
| fix | Bug 修复 | fix(BUG-001): 修复部署验证脚本 |
| docs | 文档更新 | docs: 更新任务看板 |
| refactor | 重构 | refactor: 优化缓存服务 |
| test | 测试 | test: 添加缓存测试脚本 |
| chore | 构建/工具 | chore: 更新依赖 |

## ✅ 汇报模板

### 任务完成汇报
```markdown
## ✅ [任务 ID] [任务名称] - 完成

**交付物**:
- [文件 1]
- [文件 2]

**Git 提交**: `commit_hash`

**测试结果**: ✅ 通过

**下一步**: [后续工作]
```

### 进度汇报
```markdown
## 📊 [任务 ID] [任务名称] - 进度

**核查结果**:
- [x] 任务看板：🟡 进行中
- [x] git 提交：✅ 找到提交 `xxx`
- [x] 代码文件：✅ 文件存在
- [x] 文档同步：✅ 已更新

**当前进度**: 50%
**预计完成**: HH:MM
**阻塞问题**: 无
```
